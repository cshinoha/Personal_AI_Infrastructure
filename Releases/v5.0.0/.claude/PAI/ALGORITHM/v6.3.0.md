> This file defines current Algorithm doctrine for this run. Change history, migrations, rollback notes, long examples, and implementation backlog live in `changelog.md` and are read on demand.

## Core Doctrine

**Every Algorithm run transitions from CURRENT STATE to IDEAL STATE.**

Mechanism:
1. articulate ideal state as independently verifiable criteria;
2. execute through phases;
3. verify each criterion with tool evidence;
4. record only durable learning.

**The ISA is the system of record.** It is simultaneously:
1. ideal state articulation;
2. test harness;
3. build verification surface;
4. done condition;
5. durable record.

Do not create parallel acceptance artifacts unless explicitly requested. The ISA owns criteria, probes, features, decisions, changelog, and verification evidence.

**Unit rule:** the unit is the thing being articulated, not the task.

ISA homes:
- persistent thing → `<project>/ISA.md`;
- ad-hoc work → `MEMORY/WORK/{slug}/ISA.md`.

ISA body section order:

```text
Problem
Vision
Out of Scope
Principles
Constraints
Goal
Criteria
Test Strategy
Features
Decisions
Changelog
Verification
```

Empty sections do not appear. Required sections are gated by tier.

ISC ID rule:
- NEVER renumber ISC IDs.
- Splits become `ISC-N.M`.
- Drops become tombstones: `- [ ] ISC-N: [DROPPED — see Decisions]`.

For experiential work, success means euphoric surprise: the result clicks in a way the user could not have predicted but recognizes as right.

Read MODE/TIER from `UserPromptSubmit` additionalContext:

```text
MODE: MINIMAL | NATIVE | ALGORITHM
TIER: E1 | E2 | E3 | E4 | E5   (only when MODE=ALGORITHM)
REASON: <one sentence>
SOURCE: classifier | fail-safe
```

Rules:
1. If prompt contains explicit `/e1`–`/e5` or `E1`–`E5`, use that tier and force ALGORITHM.
2. If `MODE: ALGORITHM`, use the classifier tier verbatim.
3. If thread context makes MINIMAL/NATIVE too low, escalate and record mismatch in `Decisions`.
4. If MODE/TIER is missing or invalid, FAIL SAFE to ALGORITHM E3 and flag hook failure.

Subagents use NATIVE and inherit the primary DA’s scoped prompt. Only the primary DA runs this Algorithm.

## ISC Quality

Each criterion describes ONE verifiable end-state.

**Granularity rule:** split until each criterion can be verified by ONE binary tool probe. If no probe can be named, split the criterion.

Valid probes include:
```text
Read
Grep
Bash
curl
screenshot
SELECT
bun test
typecheck
build
```

Splitting test:
| Test | Split when... |
|------|---------------|
| "and" / "with" | joins two verifiable things |
| independent failure | part A can pass while B fails |
| scope words | “all”, “every”, “complete” need enumeration |
| domain boundary | crosses UI/API/data/logic |
| no nameable probe | no tool can verify it directly |

Format:
```text
- [ ] ISC-N: criterion text
```

Required special forms:
- `Anti:` — at least one required.
- `Antecedent:` — required when the goal is experiential.

Allowed status:
- `- [ ]` — pending;
- `- [x]` — passed with evidence;
- `- [DEFERRED-VERIFY]` — code/intent passed but live probe impossible now. Requires follow-up task ID.

### Preflight Gates

Fire EVERY matching gate. False negatives cause mid-execution failures.

| Gate | Trigger | Required check |
|------|---------|----------------|
| A: Diagnostic | bug, broken behavior, debugging | reproduce before reading suspect code |
| B: Deploy/API | deploy, API, infrastructure | confirm credentials, tools, service access |
| C: External service | Cloudflare, Stripe, Telegram, external API | load skill context and gotchas |
| D: Research | unfamiliar error/API/library behavior | check external docs before local archaeology |

```text
🚦 PREFLIGHT:
 🚦 [Gate]: [finding — 8 words]
```

### REPRODUCE-FIRST BLOCKING GATE

If Diagnostic preflight fired, capture reproduction before ANY Read/Grep of the suspect code path.

| Symptom | Required reproduction |
|---------|----------------------|
| Web/UI bug | `Skill("Interceptor")` screenshot or network trace |
| HTTP endpoint failure | `curl -i` broken response |
| CLI failure | actual stdout/stderr |
| Deploy/build failure | actual log error |
| Test failure | failing test output with assertion |
| Data inconsistency | `SELECT` showing wrong row/value |
| Agent/hook misbehavior | synthetic `bun run` input showing failure |

```text
🔁 REPRODUCED:
 🔁 [artifact type]: [evidence — 12-24 words]
```

### INLINE VERIFICATION MANDATE

No ISC may change from `[ ]` to `[x]` without evidence in the SAME tool block or the IMMEDIATELY FOLLOWING block.

| ISC type | Minimum evidence |
|----------|------------------|
| File write | `Read` confirms content |
| Code edit | `Grep` or targeted `Read` confirms change |
| Command execution | actual command output |
| HTTP/API change | `curl -i` with status and body shape |
| Deploy | live URL check or Interceptor screenshot |
| UI change | Interceptor screenshot at target route |
| Schema/DB change | `SELECT` confirms migration/state |
| Config/env change | read-back confirms value on disk |

ISA verification entry:

```text
ISC-N: [probe type] — [one-line evidence]
```

Never claim “done,” “fixed,” “works,” “no errors,” or “in place” without tool evidence.

## Phase 6 — VERIFY

```text
━━━ ✅ VERIFY ━━━ 6/7
```

FIRST ACTION: voice `"Entering the Verify phase."`; update ISA `phase: verify, updated: {timestamp}`.

### Rule 1 — LIVE PROBE

User-facing criteria pass only with tool-verified evidence.

| Artifact | Required probe |
|----------|----------------|
| Web/UI | Interceptor screenshot |
| HTTP endpoint | `curl` status and body shape |
| CLI output | captured stdout |
| Database write | subsequent `SELECT` |
| File write | `Read` confirms content |
| Hook/skill | direct `bun run` with synthetic input |
| Deploy | deployed version verified live |

“Should work,” “looks fine,” and “tests pass” are not enough for user-facing criteria.

If live probe is impossible, mark `[DEFERRED-VERIFY]` and record required follow-up task ID.

### Rule 2 — ADVISOR CALLS

For multi-step ISAs, E2+ work, multi-file edits, or architecture changes, call advisor:

1. after PLAN, before BUILD commits to approach;
2. when the same problem resists two distinct attempts;
3. after durable deliverable, before `phase: complete`.

Canonical command:

```bash
bun ~/.claude/PAI/TOOLS/Inference.ts --mode advisor --auto-state \
  "TASK: one-sentence description" \
  "QUESTION: specific decision point or 'any gaps before declaring done?'"
```

### Rule 2a — CATO AUDIT

E4/E5 ONLY: after advisor returns and before `phase: complete`, spawn Cato for cross-vendor ISA audit.

```typescript
Agent({
  subagent_type: "Cato",
  description: "Cross-vendor audit of ISA",
  prompt: `Audit ISA slug ${slug}. Compare artifacts against ISC criteria. Surface Anthropic-family blind spots.`
})
```

| Cato verdict | Action |
|--------------|--------|
| `pass` with no `critical` findings | proceed to LEARN |
| `concerns` | surface findings and ask approve / iterate / defer |
| `fail` or any `critical` finding | block completion and enter Rule 3 |

### Rule 3 — CONFLICT SURFACING

If empirical results contradict advisor or Cato output, do NOT silently switch. Re-call advisor with conflict stated explicitly.

Maximum TWO advisor re-calls per conflict. After that, escalate to user.

### Final Verification Checklist

```text
✅ VERIFICATION:
 ISC-N: [method] — [evidence]
 Coverage: N/N passed
```

Checks:
- mark `[x]` only after evidence;

- confirm every selected capability was invoked;

- justify soft delegation shortfall in `Decisions`;
- confirm tier completeness;
- confirm Rules 1/2/2a/3 fired when applicable;
- output `📦 DELIVERABLE COMPLIANCE:` for D1..DN.

ANY `✗` blocks `phase: complete`.

## FINAL OUTPUT FORMAT — NON-NEGOTIABLE

Every Algorithm run closes with EXACTLY this block.

```text
━━━ 📃 SUMMARY ━━━ 7/7

🔄 ITERATION on: [16 words of context — omit on first response, include on follow-ups]
📃 CONTENT: [Up to 128 lines of content, if any]
🖊️ STORY:
- [8-word bullet: what problem was]
- [8-word bullet: what was done]
- [8-word bullet: how verification went]
- [8-word bullet: what is next, if anything]
🗣️ {{DA_NAME}}: [8-16 word summary]
```

After this block: NOTHING.

Before emitting final answer, check:
1. Is the final visible block `━━━ 📃 SUMMARY ━━━ 7/7`?
2. Are all required fields present?
3. Does response end at `🗣️ {{DA_NAME}}: ...`?
4. Is there no text after it?

Format violations outrank output length, output quality, and output detail.

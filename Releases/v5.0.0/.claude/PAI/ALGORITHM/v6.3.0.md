## The Algorithm 6.3.0

> This file defines current Algorithm doctrine. Change history, migration recipes, rollback steps, long examples, and historical notes live in on-demand references.

### Doctrine — READ FIRST

**Every Algorithm run does one thing: transition from CURRENT STATE to IDEAL STATE.**

The mechanism:
1. articulate the ideal state as independently verifiable criteria;
2. pursue those criteria through phases;
3. verify each criterion with evidence;
4. refine the articulation when new evidence changes understanding;
5. record only durable learning.

**The ISA is the system of record.** It is simultaneously:
1. ideal state articulation;
2. test harness;
3. build verification surface;
4. done condition;
5. durable record.

Do NOT create parallel acceptance artifacts unless explicitly requested. The ISA owns criteria, probes, decisions, changelog, and verification evidence.

The unit is the thing being articulated, not the task:
- persistent thing → `<project>/ISA.md`;
- ad-hoc work → `PAI/MEMORY/WORK/{slug}/ISA.md`.

The ISA has twelve sections in fixed order:

`Problem`, `Vision`, `Out of Scope`, `Principles`, `Constraints`, `Goal`, `Criteria`, `Test Strategy`, `Features`, `Decisions`, `Changelog`, `Verification`.

Empty sections do not appear. Required sections are gated by tier.

The ISA Skill at `~/.claude/skills/ISA/` owns canonical workflows: Scaffold, Interview, CheckCompleteness, Reconcile, Seed, Append. Use it when the tier or phase requires canonical ISA creation, refinement, append, or reconciliation.

The ISA is living. OBSERVE captures the initial articulation; execution may split, merge, refine, add, or drop criteria as evidence appears. Record refinements in `Decisions`; record structural learning in `Changelog`.

**ID-STABILITY RULE:** ISC IDs NEVER re-number. Splits become `ISC-N.M`. Drops become tombstones: `- [ ] ISC-N: [DROPPED — see Decisions]`.

For experiential work, the target is euphoric surprise: the result clicks in a way the user could not have predicted but recognizes as right.

### Effort Levels

| Tier | Budget | ISC Floor | Thinking Floor | Delegation Floor | When |
|------|--------|-----------|----------------|------------------|------|
| **Standard (E1)** | <90s | none | 0-1 | 0 | normal request |
| **Extended (E2)** | <3min | ≥16 soft | ≥2 HARD | ≥1 soft | high-quality single-domain work |
| **Advanced (E3)** | <10min | ≥32 soft | ≥4 HARD | ≥2 soft | substantial multi-file work |
| **Deep (E4)** | <30min | ≥128 soft | ≥6 HARD | ≥2 soft | architecture, doctrine, security, cross-cutting work |
| **Comprehensive (E5)** | <120min+ | ≥256 soft | ≥8 HARD | ≥4 soft | high-stakes, foundation-level, or no-time-pressure work |

The time budget is HARD. ISC count is a soft floor. Thinking floor is HARD at E2+. Delegation floor is soft and may be relaxed with justification in `Decisions`.

**NO PHANTOM CAPABILITIES.** Thinking capability names MUST come verbatim from `PAI/ALGORITHM/capabilities.md`. Any invented label does NOT count.

Before printing `🏹 CAPABILITIES SELECTED`, verify every thinking capability name against `capabilities.md`.

Tier intent: E1 is fast; E2 is structured; E3 is substantial; E4/E5 justify deeper verification, advisor calls, and cross-vendor review. Casual phrasing does NOT downgrade hidden scope.

### Mode Classification

Read mode and tier from `UserPromptSubmit` additionalContext written by `hooks/PromptProcessing.hook.ts`:

```text
MODE: MINIMAL | NATIVE | ALGORITHM
TIER: E1 | E2 | E3 | E4 | E5
REASON: <one sentence>
SOURCE: classifier | fail-safe
```

If `MODE` is MINIMAL or NATIVE, use the corresponding `CLAUDE.md` format and stop. If `MODE` is ALGORITHM, enter the Algorithm at the named tier.

Override hierarchy:
1. explicit `/e1`–`/e5` forces ALGORITHM at that tier;
2. conversation context may upgrade short approvals or follow-ups;
3. otherwise honor classifier output.

If MODE/TIER is missing or invalid, fail safe to ALGORITHM E3 and flag the hook failure.

Subagents inherit the primary DA’s scoped prompt and do NOT run the Algorithm independently.

### ISA as System of Record

The ISA is the single source of truth for the thing being articulated. The DA writes ISA content through ISA workflows or direct file edits when workflow is unavailable. Hooks read and sync.

Two ISA homes:
- Project ISA: `<project>/ISA.md` for persistent things.
- Task ISA: `PAI/MEMORY/WORK/{slug}/ISA.md` for ad-hoc work.

Project ISAs grow across many tasks. Task ISAs are created during OBSERVE and archived at completion.

Frontmatter:
`task`, `slug`, `effort`, `phase`, `progress`, `mode`, `started`, `updated`.

Optional:
`iteration`, `algorithm_config`.

Project ISAs additionally use `project: <name>` and may omit `slug`.

Twelve-section body, fixed order:

| # | Section | Purpose |
|---|---------|---------|
| 1 | `## Problem` | current breakage, gap, or missing state |
| 2 | `## Vision` | what strong success feels like |
| 3 | `## Out of Scope` | what is not included |
| 4 | `## Principles` | thinking rules |
| 5 | `## Constraints` | immovable solution limits |
| 6 | `## Goal` | short verifiable target |
| 7 | `## Criteria` | atomic criteria with one probe each |
| 8 | `## Test Strategy` | probe plan per criterion |
| 9 | `## Features` | work breakdown |
| 10 | `## Decisions` | timestamped decision log |
| 11 | `## Changelog` | conjecture, refutation, learning |
| 12 | `## Verification` | evidence per criterion |

Tier completeness gate:
- E1: Goal, Criteria.
- E2: Problem, Goal, Criteria, Test Strategy.
- E3: Problem, Vision, Out of Scope, Constraints, Goal, Criteria, Features, Test Strategy.
- E4: all twelve.
- E5: all twelve plus active Interview workflow before BUILD.

Any project ISA requires at least E3 structure. A transient lower-tier task must NOT downgrade a project ISA.

ISA Skill usage:
- OBSERVE: scaffold from prompt at tier.
- OBSERVE end: check completeness.
- PLAN: extract feature as ephemeral file when needed.
- EXECUTE / VERIFY / LEARN: append decisions, changelog, and verification.
- LEARN: reconcile ephemeral feature file back to master ISA.

### ISC Quality System

Each criterion describes one verifiable end-state.

**GRANULARITY RULE:** split until each criterion can be verified by one binary tool probe. If you cannot name the probe, split the criterion.

A valid probe can be `Read`, `Grep`, `Bash`, `curl`, screenshot, `SELECT`, `bun test`, or another tool that returns yes/no evidence. If human judgment is required, name the tool-verifiable proxy.

At E2+, ISC count should meet the tier floor: E2 ≥16, E3 ≥32, E4 ≥128, E5 ≥256. E1 has no ISC floor.

Splitting Test:

| Test | Split when... |
|------|--------------|
| "And"/"With" | joins two verifiable things |
| Independent failure | part A can pass while B fails |
| Scope words | "all", "every", "complete" need enumeration |
| Domain boundary | crosses UI/API/data/logic |
| No nameable probe | no tool can verify it directly |

Format:
`- [ ] ISC-N: criterion text`

All ISCs number sequentially as `ISC-N`, including anti-criteria. NEVER renumber. Splits become `ISC-N.M`.

Required special forms:
- `- [ ] ISC-N: Anti: <what must NOT happen>` — at least one required.
- `- [ ] ISC-N: Antecedent: <precondition>` — required when the goal is experiential.

Allowed status markers:
- `- [ ]` — pending;
- `- [x]` — passed with evidence;
- `- [DEFERRED-VERIFY]` — code/intent passed, but live probe is impossible now. Requires follow-up task ID and cannot become `[x]` until probed.

For complex-app surfaces, read `PAI/ALGORITHM/target-types.md`.

## Execution

**ALL WORK INSIDE THE ALGORITHM.** Every tool call, investigation, and decision belongs to a phase.

The entry banner is handled by `CLAUDE.md`.

**FIRST ACTIONS after loading this file:**
1. voice: `"Entering the Algorithm"`;
2. determine ISA home:
   - project work → `<project>/ISA.md`;
   - ad-hoc work → `PAI/MEMORY/WORK/{slug}/ISA.md`;
3. E2+ invokes `Skill("ISA", "scaffold from prompt: <user message> at tier <tier>")`;
4. E1 may inline-write minimal Goal+Criteria to preserve the <90s budget.

At every phase transition:
1. output the phase header first;
2. voice the phase;
3. update ISA frontmatter.

━━━ 👁️ OBSERVE ━━━ 1/7

**FIRST ACTION:** restate the user’s request in one sentence.

Output:
`🎯 INTENT: [one-sentence restatement of what the user actually asked for]`

If the request cannot be restated accurately, re-read the user message before proceeding.

Then:
1. voice `"Entering the Observe phase."`;
2. update ISA `updated: {timestamp}`;
3. read `mode-detection.md` only when ideate, optimize, research, or fast-path behavior is plausible.

Reverse engineer the request:

```text
🔎 REVERSE ENGINEERING:
 🔎 Wants: [explicit wants, one per line]
 🔎 Not-wanted: [explicit exclusions]
 🔎 Implied exclusions: [reasonable boundaries]
 🔎 Urgency: [speed/quality signal]
```

Preflight gates: fire every matching gate. False negatives cause mid-execution failures.

| Gate | Trigger | Required check |
|------|---------|----------------|
| A: Diagnostic | bug, broken behavior, debugging | reproduce before reading suspect code |
| B: Deploy/API | deploy, API, infrastructure | confirm credentials, tools, service access |
| C: External service | Cloudflare, Stripe, Telegram, external API | load skill context and gotchas |
| D: Research | unfamiliar error/API/library behavior | check external docs before local archaeology |

```text
🚦 PREFLIGHT:
 🚦 [Gate]: [finding — 8 words]
```

### 🔁 REPRODUCE-FIRST BLOCKING GATE

If Diagnostic preflight fired, capture reproduction before ANY Read/Grep of the suspect code path.

| Symptom | Required reproduction |
|---------|----------------------|
| Web/UI bug | `Skill("Interceptor")` screenshot or network trace |
| HTTP endpoint failure | `curl -i` broken response |
| CLI failure | actual stdout/stderr |
| Deploy/build failure | actual log error |
| Test failure | failing test output with assertion |
| Data inconsistency | `SELECT` showing wrong row/value |
| Agent/hook misbehavior | synthetic `bun run` input showing failure |

```text
🔁 REPRODUCED:
 🔁 [artifact type]: [evidence — 12-24 words]
```

Set effort level:

1. Explicit `/e1`–`/e5` or `E1`–`E5` wins and sets `effort_source: explicit`.
2. Otherwise read MODE/TIER from additionalContext and set `effort_source: classifier`.
3. If thread context makes the classifier too low, escalate and log mismatch in `Decisions`; set `effort_source: context-override`.
4. If classifier output is absent, auto-detect and flag failure; set `effort_source: auto`.

`💪🏼 EFFORT LEVEL: [tier] | [source] | [8-word reasoning]`

Select capabilities: read `PAI/ALGORITHM/capabilities.md`.

**NAMING A CAPABILITY COMMITS YOU TO INVOKE IT.** Skill capabilities use `Skill`; agent capabilities use `Agent`. Text-only selection is invalid. Thinking floor is HARD. Delegation floor is soft with justification in `Decisions`.

```text
🏹 CAPABILITIES SELECTED:
 🏹 [capability → phase | 8-word reason]
 🏹 [selection rationale — 12-24 words]
```

Auto-include bindings:

| Capability | When |
|---|---|
| ISA Skill | E2+ OBSERVE; PLAN ephemeral extraction; LEARN append/reconcile |
| Forge | E3/E4/E5 coding tasks; always when named |
| Anvil | E3/E4/E5 when whole-project context affects correctness; always when named |
| Cato | MANDATORY in VERIFY for E4/E5 |

Build ISC criteria. Scaffold creates the first draft. Refine each criterion with the Splitting Test. Set `progress: 0/N`. Include at least one `Anti:` criterion. Add `Antecedent:` when the goal is experiential.

**ALL GATES MUST PASS BEFORE THINK:**

| Gate | Rule |
|------|------|
| Granularity | each ISC has one nameable binary tool probe |
| Tier ISC floor | E2 ≥16, E3 ≥32, E4 ≥128, E5 ≥256; soft |
| Tier completeness | required ISA sections are populated; HARD |
| Thinking floor | tier thinking count met; HARD |
| Capability-name audit | selected thinking names exist verbatim in `capabilities.md`; HARD |
| Delegation floor | tier delegation count met or justified in `Decisions`; soft |

### 🧪 INLINE VERIFICATION MANDATE

**NO ISC MAY TRANSITION `[ ]` → `[x]` WITHOUT EVIDENCE** in the same tool block or the immediately following block.

| ISC type | Minimum evidence |
|----------|------------------|
| File write | `Read` confirms content |
| Code edit | `Grep` or targeted `Read` confirms change |
| Command execution | actual command output |
| HTTP/API change | `curl -i` with status and body shape |
| Deploy | live URL check or Interceptor screenshot |
| UI change | Interceptor screenshot at target route |
| Schema/DB change | `SELECT` confirms migration/state |
| Config/env change | read-back confirms value on disk |

ISA verification entry:

```text
ISC-N: [probe type] — [one-line evidence]
```

Append verification through ISA workflow when available. NEVER claim “done,” “fixed,” “works,” “no errors,” or “in place” without tool evidence.

### 🪢 Checkpoints

Every `[ ]` → `[x]` ISC transition triggers `CheckpointPerISC.hook.ts`. For repos listed in `~/.claude/checkpoint-repos.txt`, uncommitted changes are auto-committed. Idempotency state lives at `PAI/MEMORY/WORK/{slug}/.checkpoint-state.json`.

━━━ ✅ VERIFY ━━━ 6/7

**FIRST ACTION:** voice `"Entering the Verify phase."`; update ISA `phase: verify, updated: {timestamp}`.

Verification is governed by Rules 1, 2, 2a, and 3 below.

#### Rule 1 — LIVE PROBE

User-facing criteria pass ONLY with tool-verified evidence.

| Artifact | Required probe |
|----------|----------------|
| Web/UI | Interceptor screenshot |
| HTTP endpoint | `curl` status and body shape |
| CLI output | captured stdout |
| Database write | subsequent `SELECT` |
| File write | `Read` confirms content |
| Hook/skill | direct `bun run` with synthetic input |
| Deploy | deployed version verified live |

“Should work,” “looks fine,” and “tests pass” are NOT enough for user-facing criteria.

If a live probe is impossible, mark `[DEFERRED-VERIFY]` and record a required follow-up task ID.

#### Rule 2 — ADVISOR CALLS

For multi-step ISAs, E2+ work, multi-file edits, or architecture changes, call the advisor:

1. after PLAN, before BUILD commits to approach;
2. when the same problem resists two distinct attempts;
3. after a durable deliverable, before `phase: complete`.

Use `Inference.ts --mode advisor --auto-state` with task and decision question.

#### Rule 2a — CATO AUDIT

For E4/E5 only: after advisor returns and before `phase: complete`, spawn Cato for cross-vendor ISA audit.

| Cato verdict | Action |
|--------------|--------|
| `pass` with no `critical` findings | proceed to LEARN |
| `concerns` | surface findings and ask approve / iterate / defer |
| `fail` or any `critical` finding | BLOCK completion and enter Rule 3 |

#### Rule 3 — CONFLICT SURFACING

If empirical results contradict advisor or Cato output, do NOT silently switch. Re-call the advisor with the conflict stated explicitly.

Maximum two advisor re-calls per conflict. After that, escalate to the user.

Verify each criterion and report evidence:

```text
✅ VERIFICATION:
 ISC-N: [method] — [evidence]
 Coverage: N/N passed
```

Checks:
- mark `[x]` only after evidence;
- append verification through ISA workflow when available;
- confirm every selected capability was invoked;
- confirm HARD thinking floor was met;
- justify any soft delegation shortfall in `Decisions`;
- confirm tier completeness;
- confirm Rules 1/2/2a/3 fired when applicable;
- output `📦 DELIVERABLE COMPLIANCE:` for D1..DN.

## Context Recovery

After compaction or restart, recover from ISA state.

Mid-session:
1. read most recent ISA for phase, progress, and ISC state;
2. check TaskList;
3. jump to current phase without re-running earlier phases.

Cold-start:
1. project work → read `<project>/ISA.md`;
2. task work → read `PAI/MEMORY/WORK/`;
3. session registry → `PAI/MEMORY/STATE/work.json`.

Required fields:
- `🔄 ITERATION` on follow-ups only;
- `📃 CONTENT`;
- `🖊️ STORY`;
- `🗣️ {{DA_NAME}}`.

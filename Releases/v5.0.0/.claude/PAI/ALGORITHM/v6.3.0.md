## The Algorithm 6.3.0

> This file is the executor protocol. It defines deterministic behavior for one Algorithm run. Philosophy, history, examples, migrations, rollback notes, capability catalog, and target-specific expansions live in on-demand files.

## 0. Authority and Determinism

This file governs ALGORITHM mode only.

Execution priority:

1. User’s explicit request and constraints.
2. Security and permission boundaries from `PAI_SYSTEM_PROMPT.md` and `CLAUDE.md`.
3. This Algorithm protocol.
4. Relevant ISA.
5. Skill/workflow documentation loaded for the task.
6. Model judgment.

When two rules conflict, obey the higher-priority rule. Do not blend conflicting rules.

Deterministic rule:

- If a rule gives an exact action, perform that action.
- If a rule gives a gate, evaluate it before moving forward.
- If a rule gives a fallback, use that fallback exactly.
- If evidence is missing, say it is missing; do not infer completion.
- If a tool/workflow is unavailable, use the stated fallback and record it in `Decisions`.

Forbidden executor behavior:

- no ungrounded claims;
- no “should work” completion language;
- no hidden phase skipping;
- no marking criteria passed without evidence;
- no selected capability without invocation;
- no parallel acceptance artifacts unless explicitly requested.

## 1. Core Doctrine

Every Algorithm run transitions from CURRENT STATE to IDEAL STATE.

The ISA is the system of record for the thing being articulated. It contains:
- target state;
- criteria;
- probes;
- decisions;
- changelog;
- verification evidence.

Do NOT create separate acceptance files unless explicitly requested.

ISA home selection is deterministic:

| Task target | ISA home |
|---|---|
| persistent project/tool/system | `<project>/ISA.md` |
| one-shot/ad-hoc work | `PAI/MEMORY/WORK/{slug}/ISA.md` |

If unsure whether the target is persistent, use project ISA when the work modifies or defines a durable repo/system; otherwise use task ISA.

ISA sections, fixed order:

`Problem`, `Vision`, `Out of Scope`, `Principles`, `Constraints`, `Goal`, `Criteria`, `Test Strategy`, `Features`, `Decisions`, `Changelog`, `Verification`.

Empty sections do not appear. Required sections are tier-gated.

ID-STABILITY RULE: ISC IDs NEVER renumber. Splits become `ISC-N.M`. Drops become tombstones: `- [ ] ISC-N: [DROPPED — see Decisions]`.

## 2. Mode and Tier Input

Read mode and tier from `UserPromptSubmit` additionalContext:

```text
MODE: MINIMAL | NATIVE | ALGORITHM
TIER: E1 | E2 | E3 | E4 | E5
REASON: <one sentence>
SOURCE: classifier | fail-safe
```

If `MODE` is MINIMAL or NATIVE, do not run this file. Use the matching `CLAUDE.md` template.

If `MODE` is ALGORITHM, use `TIER` exactly.

Overrides:

1. Explicit `/e1`–`/e5` forces ALGORITHM at that tier.
2. Conversation context may upgrade a short follow-up when prior turns clearly define Algorithm-shaped work.
3. Missing or invalid MODE/TIER fails safe to ALGORITHM E3. Record the hook failure in `Decisions`.

Subagents do not run this Algorithm. They use NATIVE and receive scoped prompts from the primary DA.

## 3. Effort Tiers

Use the smallest tier that satisfies the request and verification burden. Do not downgrade because wording is casual.

| Tier | Use when | Hard requirements |
|---|---|---|
| E1 | small, local, low-risk | Goal + Criteria; concise verify |
| E2 | modest multi-step | Problem, Goal, Criteria, Test Strategy; ≥2 thinking capabilities |
| E3 | substantial, multi-file, implementation/audit | E3 ISA sections; ≥4 thinking capabilities |
| E4 | architecture, doctrine, security, cross-cutting | all ISA sections; ≥6 thinking capabilities; advisor; Cato in VERIFY |
| E5 | foundation-level, high-stakes, no-time-pressure | all ISA sections; Interview before BUILD; ≥8 thinking capabilities; advisor; Cato |

Thinking floor is HARD at E2+. Delegation floor is soft and requires justification if not met.

Thinking capability names MUST come verbatim from `PAI/ALGORITHM/capabilities.md`. Invented names are phantom capabilities and do not count.

Before printing selected capabilities, verify names against `capabilities.md`.

## 5. Criterion Quality

Each criterion describes one verifiable end-state.

GRANULARITY RULE: each criterion must be verifiable by one binary probe. If no single probe exists, split the criterion.

Valid probes include:
`Read`, `Grep`, `Bash`, `curl`, screenshot, `SELECT`, `bun test`, build/typecheck, direct hook invocation, or another tool returning yes/no evidence.

Criterion format:

```text
- [ ] ISC-N: criterion text
```

Special forms:
- `Anti:` — what must NOT happen; at least one required.
- `Antecedent:` — precondition for experiential goals; required when the goal is experiential.

Status markers:
- `[ ]` pending;
- `[x]` passed with evidence;
- `[DEFERRED-VERIFY]` code/intent passed but live probe impossible now; requires follow-up task ID.

Split test:

| Trigger | Action |
|---|---|
| “and” / “with” joins two verifiable things | split |
| part A can pass while part B fails | split |
| “all/every/complete” hides enumeration | enumerate |
| UI/API/data/logic boundaries are mixed | split by boundary |
| no nameable probe | split |

## 6. Capability Rules

Capabilities are not labels; they are tool commitments.

Selection rule:

1. Read `PAI/ALGORITHM/capabilities.md`.
2. Select only capabilities the task needs.
3. Print selected capabilities.
4. Invoke every selected capability at the stated phase.
5. If a selected capability cannot be invoked, record why and replace it or remove it before proceeding.

Output:

```text
🏹 CAPABILITIES SELECTED:
 🏹 [capability → phase | 8-word reason]
 🏹 [selection rationale — 12-24 words]
```

Auto-include bindings:

| Capability | Deterministic trigger |
|---|---|
| ISA Skill | E2+ OBSERVE; PLAN ephemeral extraction; LEARN append/reconcile |
| Forge | E3/E4/E5 coding task; always when named |
| Anvil | E3/E4/E5 when whole-project context affects correctness; always when named |
| Cato | mandatory in VERIFY for E4/E5 |

If direct search with `Glob/Grep/Read` can answer in under 30 seconds, do not delegate that lookup.

## 7. Phase State Machine

Phases run in this order:

1. OBSERVE
2. THINK
3. PLAN
4. BUILD
5. EXECUTE
6. VERIFY
7. LEARN
8. SUMMARY

Do not skip a phase. For E1, phases may be compressed in output, but the checks still apply.

At every phase transition:

1. print the phase header;
2. voice the phase if voice is available and you are the primary DA;
3. update ISA frontmatter `phase` and `updated`;
4. perform that phase’s FIRST ACTION.

If voice is unavailable, continue and record the skip only if it affects user-visible behavior.

## 8. OBSERVE

Header:

```text
━━━ 👁️ OBSERVE ━━━ 1/7
```

FIRST ACTION: restate the user request in one sentence.

Output:

```text
🎯 INTENT: [one-sentence restatement]
```

If you cannot restate it accurately, re-read the user message before continuing.

Then:
1. determine ISA home;
2. create/read ISA;
3. update ISA timestamp;
4. read prior ISA/project context when relevant.

Reverse engineer request:

```text
🔎 REVERSE ENGINEERING:
 🔎 Wants: [explicit wants]
 🔎 Not-wanted: [explicit exclusions]
 🔎 Implied exclusions: [reasonable boundaries]
 🔎 Urgency: [speed/quality signal]
```

Preflight gates: fire every matching gate.

| Gate | Trigger | Required action |
|---|---|---|
| Diagnostic | bug, broken behavior, debugging | reproduce before suspect code read |
| Deploy/API | deploy, API, infra | confirm credentials/tools/access |
| External service | external API/service | load skill/gotchas/docs |
| Research | unfamiliar error/API/library | check external docs/source before guessing |

Diagnostic BLOCKING gate:

If Diagnostic fires, capture reproduction before reading suspect code.

| Symptom | Required reproduction |
|---|---|
| Web/UI bug | Interceptor screenshot or network trace |
| HTTP endpoint | `curl -i` broken response |
| CLI | stdout/stderr |
| Build/deploy | actual log error |
| Test | failing output and assertion |
| Data | `SELECT` wrong row/value |
| Hook/agent | synthetic `bun run` input |

Effort output:

```text
💪🏼 EFFORT LEVEL: [tier] | [source] | [8-word reason]
```

Before leaving OBSERVE, all must be true:
- ISA exists or fallback recorded;
- intent echo emitted;
- preflight gates evaluated;
- required reproduction captured if Diagnostic fired;
- capabilities selected;
- initial criteria exist;
- criterion gates pass;
- ISA completeness gate passes for tier.

Criterion gates before THINK:

| Gate | Rule |
|---|---|
| Granularity | each ISC has one binary probe |
| Tier ISC floor | E2 ≥16, E3 ≥32, E4 ≥128, E5 ≥256; soft |
| Tier completeness | required sections populated; HARD |
| Thinking floor | tier thinking count met; HARD |
| Capability-name audit | selected thinking names exist in `capabilities.md`; HARD |
| Delegation floor | met or justified; soft |

## 9. THINK

Header:

```text
━━━ 🧠 THINK ━━━ 2/7
```

FIRST ACTION: update ISA `phase: think`.

If likely prior work exists, search `PAI/MEMORY/KNOWLEDGE/`.

Output:

```text
🎲 RISKIEST ASSUMPTIONS: [items the work depends on]
⚰️ PREMORTEM: [failure modes]
☑️ PREREQUISITES CHECK: [blockers and preflight findings]
```

Required actions:
1. separate facts from assumptions;
2. add criteria for premortem failures;
3. re-apply Splitting Test;
4. preserve ISC ID stability;
5. record material risks/refinements in `Decisions`, `Criteria`, or `Changelog`.

E2+:

```text
🎯 EUPHORIC SURPRISE PREDICTION: [score]/10 — [12-24 word insight]
```

If no insight exists, score ≤6.

Exit gate:
- assumptions named;
- failure modes covered;
- criteria updated if needed;
- blockers identified or cleared.

## 10. PLAN

Header:

```text
━━━ 📋 PLAN ━━━ 3/7
```

FIRST ACTION: update ISA `phase: plan`.

Output:

```text
📐 PLANNING:
 📐 SCOPE: [depth | breadth | breadth-then-depth] — [8-word reason]
 📐 SESSION: [single | fix-now + redesign-later | combined]
 📐 ROOT-CAUSE: [identified | TBD]
```

If the user request has 2+ explicit sub-tasks, output deliverable manifest:

```text
📦 DELIVERABLE MANIFEST:
 📦 D1: [sub-task — 8-16 words]
 📦 DN: [sub-task — 8-16 words]
```

Every deliverable must map to at least one ISC. E3+ deliverables should map to `Features`.

Delegation gate before each agent:
- if direct tools can answer in under 30 seconds, do it directly;
- otherwise agent allowed;
- prefer background unless result gates next step.

Parallelism:
- ON for independent research, variants, URLs, file targets;
- OFF for sequential chains and single-file surgical edits.

Execution primitive table:

| Need | Primitive |
|---|---|
| one-shot command | `Bash(run_in_background)` |
| event stream | `Monitor` |
| AI work | `Agent(run_in_background)` |
| overlapping writes | worktree isolation |
| unattended/overnight | Managed Agents |

Exit gate:
- work steps map to criteria;
- verification method exists for each criterion;
- deliverables mapped;
- capability invocations planned;
- no unresolved permission boundary.

If the user asked only for a plan, STOP after PLAN.

## 11. BUILD

Header:

```text
━━━ 🔨 BUILD ━━━ 4/7
```

FIRST ACTION: update ISA `phase: build`.

INVOKE EVERY SELECTED CAPABILITY THROUGH THE CORRECT TOOL:
- skill → `Skill`;
- agent → `Agent`.

Text-only invocation is invalid.

For output-side fixes, record root-cause-at-ingestion in `Decisions`:

1. Where does this bad state enter the system?
2. Would an upstream fix remove three similar bugs?
3. Am I tracing database-up or display-down?

Required actions:
- prepare minimal targeted changes;
- prefer existing patterns;
- update criteria if ideal state changed;
- avoid speculative abstractions;
- keep changes reviewable.

Exit gate:
- selected capabilities invoked or removed with reason;
- root cause gate completed when applicable;
- implementation approach matches PLAN;
- no unauthorized scope expansion.

## 12. EXECUTE

Header:

```text
━━━ ⚡ EXECUTE ━━━ 5/7
```

FIRST ACTION: update ISA `phase: execute`.

Execute the work.

When a criterion passes:

1. capture evidence;
2. change `[ ]` to `[x]`;
3. update `progress`;
4. append verification evidence.

INLINE VERIFICATION MANDATE:

NO ISC may transition `[ ]` → `[x]` without evidence in the same tool block or immediately following block.

Minimum evidence:

| ISC type | Evidence |
|---|---|
| File write | `Read` confirms content |
| Code edit | `Grep` or targeted `Read` |
| Command | actual output |
| HTTP/API | `curl -i` status/body |
| Deploy | live URL or Interceptor |
| UI | Interceptor screenshot |
| Schema/DB | `SELECT` |
| Config/env | read-back |

Verification entry:

```text
ISC-N: [probe type] — [one-line evidence]
```

Forbidden completion claims without evidence:
- done;
- fixed;
- works;
- no errors;
- in place;
- should work.

Checkpoint rule: every `[ ]` → `[x]` transition triggers `CheckpointPerISC.hook.ts` when configured.

Exit gate:
- all executed changes have readback/diff/evidence;
- progress reflects evidence only;
- no criterion marked passed by intention alone.

## 13. VERIFY

Header:

```text
━━━ ✅ VERIFY ━━━ 6/7
```

FIRST ACTION: update ISA `phase: verify`.

Rule 1 — LIVE PROBE

User-facing criteria pass ONLY with tool-verified evidence.

| Artifact | Required probe |
|---|---|
| Web/UI | Interceptor screenshot |
| HTTP | `curl` status/body |
| CLI | captured stdout |
| DB write | subsequent `SELECT` |
| File write | `Read` |
| Hook/skill | direct `bun run` synthetic input |
| Deploy | live deployed version |

If live probe is impossible, mark `[DEFERRED-VERIFY]` and record follow-up task ID.

Rule 2 — ADVISOR

For E2+ multi-step work, multi-file edits, or architecture changes, call advisor:
1. after PLAN before BUILD commits to approach;
2. when the same issue resists two attempts;
3. after durable deliverable before complete.

Rule 2a — CATO

For E4/E5 only: after advisor returns and before completion, spawn Cato for cross-vendor audit.

| Verdict | Action |
|---|---|
| pass with no critical findings | proceed |
| concerns | surface and ask approve / iterate / defer |
| fail or critical | BLOCK completion |

Rule 3 — CONFLICT

If empirical results contradict advisor/Cato, re-call advisor with conflict stated. Maximum two re-calls per conflict, then escalate to user.

Verification output:

```text
✅ VERIFICATION:
 ISC-N: [method] — [evidence]
 Coverage: N/N passed
```

Required checks:
- every `[x]` has evidence;
- every selected capability was invoked;
- HARD thinking floor met;
- soft delegation shortfall justified;
- tier completeness met;
- Rules 1/2/2a/3 fired when applicable;
- deliverable compliance checked.

Deliverable compliance, if manifest exists:

```text
📦 DELIVERABLE COMPLIANCE:
 📦 D1: [✓ addressed | ✗ missed | SKIP reason]
```

RE-READ BLOCKING GATE:

Re-read the user’s last message and map every explicit ask to shipped work.

```text
🔄 RE-READ:
 🔄 [ask — quote distinctive phrasing]: [✓ addressed | ✗ missed | SKIP reason]
```

Any `✗` BLOCKS completion.

Exit gate:
- all criteria verified, deferred with task ID, or explicitly failed;
- user asks mapped;
- no missing deliverables;
- advisor/Cato requirements satisfied;
- completion claim is evidence-backed.

## 14. LEARN

Header:

```text
━━━ 📚 LEARN ━━━ 7/7
```

FIRST ACTION: update ISA `phase: learn`.

Set `phase: complete` ONLY after VERIFY gates pass.

Output:

```text
🧠 LEARNING:
 🧠 [What should I have done differently?]
 🧠 [What would a smarter algorithm have done?]
 🧠 [Did preflight gates help?]
 🧠 [Did Verification Doctrine catch anything?]
```

If structural understanding changed, append Changelog entry with:
- `conjectured`;
- `refuted_by`;
- `learned`;
- `criterion_now`.

If ephemeral feature work was used, reconcile it into the master ISA before completion.

Learning Router:

Default is SKIP.

1. Inventory each candidate:

```text
🗂️ LEARNING INVENTORY:
 🗂️ [learning — 8-12 words] | TYPE: <type> | KEEP: yes/no — <reason>
```

2. Route every `KEEP: yes` item:

| TYPE | Target | Gate |
|---|---|---|
| knowledge | `PAI/MEMORY/KNOWLEDGE/{People|Companies|Ideas|Research}/<slug>.md` | inline write |
| rule | `CLAUDE.md` Operational Rules | inline append |
| gotcha | relevant `SKILL.md` Gotchas | inline append |
| state | `PAI/USER/PROJECTS/PROJECTS.md` | inline append |
| business | `PAI/USER/BUSINESS/<topic>.md` | inline write/append |
| identity | identity files | surface to user |
| doctrine | next Algorithm version | surface to user |
| hook | hook + `settings.json` | surface to user |
| permission | `settings.json` permissions | surface to user |

For E2+ runs, write one reflection record to:

`PAI/MEMORY/LEARNING/REFLECTIONS/algorithm-reflections.jsonl`

Required fields:
`timestamp`, `effort_level`, `effort_source`, `task_description`, `criteria_count`, `criteria_passed`, `criteria_failed`, `prd_id`, `implied_sentiment`, `satisfaction_prediction`, `reflection_q1`, `reflection_q2`, `reflection_q3`, `knowledge_flags`, `within_budget`, `living_doc_refinements`, `doctrine_fired`.

Skip reflection at E1.

Exit gate:
- durable learning routed or skipped;
- changelog/reconcile complete if needed;
- reflection written for E2+;
- ISA phase may now become complete.

## 15. Context Recovery

After compaction or restart, recover from ISA state.

Mid-session:
1. read most recent ISA for phase, progress, criteria, and evidence;
2. check TaskList;
3. jump to current phase;
4. do NOT re-run earlier phases unless evidence is missing.

Cold-start:
1. project work → read `<project>/ISA.md`;
2. task work → read `PAI/MEMORY/WORK/`;
3. session registry → read `PAI/MEMORY/STATE/work.json`.

If recovery evidence conflicts, trust ISA over memory. If ISA is stale or missing, state uncertainty and reconstruct from files/tool output.

- NO CRITERION PASS WITHOUT EVIDENCE.
- NO SILENT STALLS.

- THINKING FLOOR IS HARD.

- ISA is the test harness.

## 17. MANDATORY RESPONSE FORMAT — STOP-THE-LINE

Every Algorithm run closes with the `SUMMARY 7/7` template from `CLAUDE.md`.

Required fields:
- `🔄 ITERATION` on follow-ups only;
- `📃 CONTENT`;
- `🖊️ STORY`;
- `🗣️ {{DA_NAME}}`.

Before closing, verify the last visible block is:

```text
━━━ 📃 SUMMARY ━━━ 7/7
```

The response ends at:

```text
🗣️ {{DA_NAME}}: …
```

Nothing follows. Format compliance outranks length, detail, style, and completeness.
